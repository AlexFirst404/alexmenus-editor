# Продвинутые настройки меню

Ключи верхнего уровня файла меню (`menus/*.yml`), помимо базовых из [Menu-Types](Menu-Types.md). Все —
**необязательны**. Здесь: живое обновление плейсхолдеров (`update-interval`), аргументы команды (`args`),
предмет-открывашка (`open-item`) и анимация открытия (`open-animation`).

| Ключ | Тип | Кратко |
|---|---|---|
| `update-interval` | число (тики) | Живая перерисовка открытого меню каждые N тиков. `0` = выкл. |
| `args` | список имён | Захват аргументов команды открытия в `{имя}` / `{argN}` / `{args}`. |
| `open-item` | секция | «Предмет-открывашка» в хотбаре (ПКМ открывает меню). |
| `open-animation` | секция | Пошаговое появление предметов при открытии (только `chest`). |

---

## `update-interval` — живое обновление плейсхолдеров

Число **тиков** между перерисовками уже **открытого** меню. Плагин пере-вычисляет `title`, `name`, `lore`,
видимость (`view-requirement`) **на месте**, не переоткрывая окно. `0` (или отсутствует) — без живых
обновлений. Работает и для `chest`, и для `inventory`-меню.

```yaml
type: chest
title: "<gold>Онлайн: <white>%server_online%"
update-interval: 20      # обновлять раз в секунду (20 тиков)
items:
  '13':
    material: CLOCK
    name: "<yellow>Баланс: <green>%vault_eco_balance%"
    lore:
      - "<gray>TPS: <white>%server_tps_1%"
```

- Значение приводится к минимуму `0` (отрицательное → `0` = выкл.).
- Все живые меню обновляет **одна** общая повторяющаяся задача плагина.
- Значение — интервал перерисовки, а не «частота открытия»; на производительность влияет число открытых
  живых меню × частота. Разумные значения — `10`–`40` тиков.

## `args` — аргументы команды открытия

Список **имён** аргументов. Когда меню открывают с хвостовыми токенами (`/menu <a> <b> …`, `/am open <id>
<игрок> <a> <b> …`), плагин раскладывает их и делает доступными как подстановки `{...}` в `title`, `name`,
`lore` **и** в действиях (`command`, `text`, `title`/`subtitle`, `server`, `permission` и т.п.).

```yaml
type: chest
title: "<gold>Профиль: <yellow>{name}"
commands: [profile]           # /profile <name>
args: [name]
items:
  '13':
    material: PLAYER_HEAD
    name: "<yellow>{name}"
    lore:
      - "<gray>Открыть статистику"
    clicks:
      any:
        - type: run_command
          as: console
          command: "stats show {name}"
```

Как формируются подстановки (`buildArgs`):

| Токен | Значение |
|---|---|
| `{имя}` | Именованный аргумент из списка `args` по позиции: 1-е имя = 1-й токен, 2-е имя = 2-й токен … |
| **последнее имя** | **Захватывает весь остаток** строки (все токены с его позиции и дальше, через пробел). |
| `{arg1}`, `{arg2}`, … | Позиционные токены (1-индексация), всегда доступны, даже без `args`. |
| `{args}` | Все переданные токены одной строкой (через пробел). |

Пример «последнее имя ловит остаток»:

```yaml
commands: [broadcastmenu]     # /broadcastmenu <player> <текст сообщения...>
args: [target, text]
items:
  '11':
    material: PAPER
    name: "<green>Отправить"
    clicks:
      any:
        - type: run_command
          as: console
          command: "tell {target} {text}"      # {text} = всё после первого токена
```

Тонкости:

- Пустые/пробельные имена в списке `args` отбрасываются, порядок сохраняется.
- Если токенов меньше, чем имён, недостающие именованные подстановки — пустая строка (`""`).
- Аргументы **привязаны к текущему меню** игрока: `open_menu`/`navigate` в под-меню сбрасывает их (под-меню
  строит свои), а `back` восстанавливает те, с которыми открывалось меню, к которому вернулись.
- Подстановка `{...}` идёт **до** PlaceholderAPI (`%...%`), так что аргумент может даже попасть внутрь
  плейсхолдера. Порядок разворачивания — [Actions](Actions.md), [Placeholders-and-Integrations](Placeholders-and-Integrations.md).

## `open-item` — предмет-открывашка (хотбар)

Секция, описывающая предмет, который **открывает это меню по ПКМ** (item-bind / «компас-меню»). Предмет
несёт id меню в PDC (тот же механизм, что у `give_item` с полем `menu` — см. [Actions](Actions.md)), поэтому
ПКМ по нему открывает меню через штатный item-триггер.

```yaml
type: chest
title: "<gold>Меню сервера"
commands: [menu]
open-item:
  material: COMPASS       # материал опенера
  name: "<gold>Меню <gray>(ПКМ)"
  cmd: "menu_compass"     # строковый custom-model-data (ресурспак), опц.
  slot: 8                 # слот хотбара 0–8; иначе/-1 = первый свободный слот
  give-on-join: true      # выдавать при входе/респавне, если у игрока его ещё нет
items:
  '13': { material: NETHER_STAR, name: "<aqua>Магазин", clicks: { any: [ { type: open_menu, menu: shop } ] } }
```

| Поле | По умолчанию | Назначение |
|---|---|---|
| `material` | `COMPASS` | Материал предмета (неизвестный/не-предмет → `COMPASS`). |
| `name` | — | Отображаемое имя (цвета/MiniMessage). |
| `cmd` | — | Строковый custom-model-data для модели из ресурспака (как `cmd` предмета, см. [Items-and-Clicks](Items-and-Clicks.md)). |
| `slot` | `-1` | Целевой слот хотбара `0–8`. Вне диапазона/`-1` → добавить в первый свободный слот. |
| `give-on-join` | `false` | `true` — выдавать при входе/респавне, если у игрока ещё нет опенера этого меню. |

Поведение выдачи (`OpenItemListener`):

- Гейтится глобальным `open-items` в `config.yml` (см. [Configuration](Configuration.md)): если он `false` —
  никакой выдачи на входе.
- Опенер выдаётся с небольшой задержкой после входа (и на респавне), **только если** его ещё нет (ищется по
  PDC-тегу меню) — дубликатов не будет.
- Если задан `slot 0–8` и он **свободен** — предмет кладётся туда; если слот **занят** реальной вещью, опенер
  добавляется в первый свободный слот (чужую вещь никогда не перезаписывает).
- Право на само открытие проверяется при ПКМ (нужны `alexmenus.use` и `permission:` меню, если задано).

> `open-item` описывает опенер, который меню раздаёт само. Разово выдать предмет-открывашку из действия можно
> через [`give_item`](Actions.md) с полем `menu`.

## `open-animation` — анимация открытия

Секция «пошагового появления» предметов при открытии **сундук-меню** (`type: chest`). Предметы
расставляются по очереди — `interval` тиков на шаг, с необязательным звуком на каждом шаге.

```yaml
type: chest
title: "<gold>Магазин"
rows: 3
open-animation:
  type: sweep            # порядок появления
  interval: 2            # тиков на шаг
  sound: "minecraft:ui.button.click"   # звук на каждом шаге, опц.
items:
  '0': { material: DIAMOND, name: "<b>1" }
  '4': { material: EMERALD, name: "<b>2" }
  '8': { material: GOLD_INGOT, name: "<b>3" }
```

| Поле | По умолчанию | Назначение |
|---|---|---|
| `type` | `none` | Порядок появления (см. таблицу ниже). |
| `interval` | `2` | Тиков между шагами (минимум `1`). |
| `sound` | — | Ключ звука (`namespace:path`), проигрываемый на каждом шаге. |

Значения `type`:

| `type` | Алиасы | Порядок появления |
|---|---|---|
| `none` | — | Без анимации — все предметы сразу (как без секции). |
| `sweep` | `slots`, `slot` | По возрастанию слота, по одному слоту за шаг. |
| `rows` | `row` | Ряд за рядом (сверху вниз), целый ряд за шаг. |
| `random` | `shuffle` | В перемешанном порядке слотов, по одному за шаг. |

Тонкости:

- Неизвестное значение `type` → `none` (анимации нет).
- Анимация применяется к **`chest`**-меню; `inventory`-меню появляются сразу (анимация к ним не применяется).
- Хорошо сочетается с `update-interval`: анимация — разовое появление при открытии, живые обновления идут
  после.
